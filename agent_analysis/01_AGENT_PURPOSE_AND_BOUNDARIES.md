# SCeQTL-Agent 项目目标、边界与核心问题分析

> **文档版本**: 1.0  
> **日期**: 2026-03-18  
> **目的**: 系统梳理Agent的定位、核心目标、关键边界与待解决的核心问题，为架构升级提供清晰的问题定义

---

## 1. 项目定位与愿景

### 1.1 一句话定义

SCeQTL-Agent 是一个**本体感知、跨库融合的人源单细胞RNA-seq元数据智能检索系统**，通过自然语言交互帮助生物信息学家快速定位、比较和获取全球分散的数据集。

### 1.2 核心价值主张

| 维度 | 现有痛点 | 我们的解决方案 |
|------|---------|--------------|
| **数据发现** | 12个主要数据库各自为政，研究者需逐个搜索 | 统一查询入口，一次检索覆盖全部数据源 |
| **语义理解** | 关键词匹配无法处理"brain → cerebral cortex"等层级关系 | 基于UBERON/MONDO/CL本体的智能语义扩展 |
| **去重融合** | 同一样本在GEO和SRA中重复出现，难以识别 | 跨库关联(entity_links) + 身份哈希去重 |
| **查询门槛** | 需要掌握各数据库的查询语法和ID体系 | 自然语言交互，零学习成本 |
| **数据评估** | 无法快速判断数据质量与可用性 | 数据质量评分 + 完整血缘追溯 |

## 2. Agent需要实现的核心目标

### 2.1 功能性目标（Functional Goals）

#### G1: 精确理解复杂查询意图

**目标描述**: Agent能够准确解析用户自然语言查询，识别其中的意图类型、生物学实体、过滤条件和隐式需求。

**具体表现**:
- 支持6种查询意图：SEARCH / COMPARE / STATISTICS / EXPLORE / DOWNLOAD / LINEAGE
- 准确提取5类生物学实体：tissue, disease, cell_type, assay, organism
- 处理中英文双语输入
- 识别否定条件（"non-cancer"）、范围条件（"adult samples"）、比较意图（"difference between"）

**当前状态**: 
- 规则引擎覆盖85%常见查询，但复杂语义（如"最近3年高引用的脑类器官研究"）仍需提升
- 评测通过率92.2%，但Ontology类别仅76%

---

#### G2: 基于本体的智能语义扩展

**目标描述**: 将用户输入的非标准化术语映射到标准本体ID，并智能扩展查询范围。

**具体表现**:
- "brain" → 扩展至 ["brain", "cerebral cortex", "hippocampus", "cerebellum", ...]
- "cancer" → 扩展至具体癌症类型 ["lung cancer", "breast cancer", ...]
- 支持umbrella term（伞形术语）的智能展开
- 跨本体映射（UBERON tissue → 数据库tissue字段实际值）

**当前状态**:
- 已集成113K本体术语（UBERON 9.5K + MONDO 30.7K + CL 16.7K + EFO 56.5K）
- 5步解析管线：exact → synonym → fuzzy → LLM → fallback
- **核心问题**: umbrella term扩展率不足，"immune cell"等类别术语无法有效展开

---

#### G3: 精确的查询范围界定

**目标描述**: 这是本项目的**核心目标**——良好理解查询并给出**准确的范围**。

**具体表现**:
| 查询示例 | 准确范围界定 |
|---------|-------------|
| "阿尔茨海默病的大脑样本" | 疾病=AD 且 组织=brain/cerebral cortex/hippocampus... |
| "10x Genomics平台的前额叶皮层数据" | assay=10x 3' v3 且 tissue=frontal cortex |
| "排除血液样本的其他组织的免疫细胞" | cell_type包含T/B/macrophage... 且 tissue≠blood |
| "CellXGene和GEO都有的项目" | 需要跨库关联(entity_links)过滤 |

**挑战点**:
- 隐式条件的推断（如"最近" → published_after = 3年前）
- 排除条件的正确处理（negated entities）
- 多条件间的AND/OR关系推断
- 数值范围的理解（"高引用" → citation_count > ?）

---

#### G4: 跨库去重与结果融合

**目标描述**: 识别并合并来自不同数据库的同一实体，提供统一视图。

**具体表现**:
- PRJNA12345 ↔ GSE67890 自动关联
- 基于identity hash的样本级去重
- 保留所有来源ID便于追溯
- 质量评分选择最佳元数据

**当前状态**:
- 9,966条跨库链接已建立
- 100K去重候选待处理
- **核心问题**: 复杂去重策略（需考虑样本ID、标题、作者、时间等多因子）

---

#### G5: 可解释的结果呈现

**目标描述**: 不仅给出结果，还要说明"为什么是这个结果"。

**具体表现**:
- 自然语言摘要："找到47个数据集，其中23个来自GEO，24个来自CellXGene..."
- 本体扩展说明："'brain' 已扩展至 cerebral cortex, hippocampus等12个相关组织"
- 数据血缘链：完整的来源追溯
- 质量评估报告：字段完整度、交叉验证分数

---

### 2.2 非功能性目标（Non-functional Goals）

#### NFG1: 成本效率

- **目标**: 85%以上查询无需LLM调用
- **现状**: 84.9%（规则68.6% + 模板16.3%）
- **手段**: 规则引擎优先、模板SQL、多层缓存

#### NFG2: 响应速度

| 查询类型 | P50目标 | P95目标 |
|---------|--------|--------|
| ID直接查询 | 50ms | 100ms |
| 简单搜索 | 100ms | 300ms |
| 复杂搜索（本体扩展） | 500ms | 1500ms |
| 统计查询 | 500ms | 2000ms |

#### NFG3: 可扩展性

- 支持新数据源的无代码/低代码接入
- Schema变更时自动适应
- 新本体术语自动同步

### 4.2 P1: 查询理解问题

#### P1.1 意图歧义消解

**问题描述**: 同一句查询可能有多种理解方式。

**示例**:
- "Alzheimer's brain samples": 
  - 理解A: disease=AD AND tissue=brain
  - 理解B: 研究AD对brain影响的样本
  - 理解C: 来自AD患者brain区域的样本

**当前方案**: 规则优先级 + LLM兜底
**待优化**: 引入更丰富的上下文消歧机制

---

#### P1.2 复杂实体抽取

**问题描述**: 复杂查询中包含嵌套或条件实体。

**示例**:
- "Find datasets with T cells in tumor but not in blood"
  - 涉及: cell_type=T cell, tissue=tumor, 否定(tissue=blood)
  - 条件逻辑: (cell_type=T cell AND tissue=tumor) AND NOT (cell_type=T cell AND tissue=blood)
  - 或者理解为: 在tumor中找到的T细胞，排除那些只在blood中的样本

**当前方案**: 简单实体列表抽取
**待优化**: 支持嵌套条件、关系抽取

---

#### P1.3 否定与排除条件

**问题描述**: 正确处理"非"、"排除"、"不含"等否定语义。

**示例**:
- "non-cancer samples" → disease ≠ cancer (但cancer有多种)
- "samples without cell type annotation" → cell_type IS NULL
- "除GEO外的数据源" → source_database ≠ GEO

**当前方案**: BioEntity.negated 标志位
**待优化**: 否定范围的精确界定（全盘否定vs部分否定）

---

### 4.3 P2: 范围界定问题（核心中的核心）

#### P2.1 本体语义扩展的精确性

**问题描述**: umbrella term的展开需要平衡召回率和精确率。

**示例**:
- "brain" 应该扩展到哪些子项？
  - 过少 → 遗漏相关数据（recall低）
  - 过多 → 引入不相关数据（precision低）
  - 理想: 根据用户意图动态调整

**当前问题**:
```
评测数据:
- Ontology类别通过率: 19/25 (76%)
- 失败案例: "immune cell" 未能有效展开到具体细胞类型
- 原因: 当前umbrella term列表是硬编码，覆盖不完整
```

**潜在解决方案**:
1. 基于本体层级深度的动态扩展策略
2. 结合数据库实际值的智能扩展（哪些子项在DB中有数据）
3. 用户反馈驱动的扩展学习

---

#### P2.2 数值范围的理解

**问题描述**: 将模糊描述转换为精确的数值过滤。

**示例**:
| 用户描述 | 可能的数值含义 | 挑战 |
|---------|--------------|------|
| "recent datasets" | published_after = 1-3年前？ | 相对时间基准 |
| "highly cited" | citation_count > 10? 50? 100? | 领域基准差异 |
| "large-scale" | n_cells > 10000? 100000? | 技术发展阶段差异 |
| "adult samples" | age_range = 18-65? | 没有统一年龄字段 |

**当前方案**: 部分硬编码规则
**待优化**: 自适应阈值学习、基于领域统计的动态设定

---

#### P2.3 隐式条件的推断

**问题描述**: 用户未明确说出但隐含的条件。

**示例**:
- "frontal cortex" → 隐含 organism=human（除非特别说明mouse）
- "Alzheimer's" → 隐含 disease=AD，但可能也关注related diseases
- "查找数据集" → 隐含 exclude_downloaded? 根据历史？

**挑战**: 避免过度推断，保持可预测性

---

### 4.4 P3: 跨库融合问题

#### P3.1 跨库实体关联的完整性

**问题描述**: 建立和维护不同数据库间的实体对应关系。

**当前状态**:
```
跨库链接统计:
- PRJNA↔GSE (same_as): 4,142
- PMID 关联: 5,756
- DOI 关联: 68
- 去重候选: 100,000 (待验证)

问题: 仅覆盖约20%的可能关联
```

**待解决**:
- 基于标题/作者/时间的相似度匹配
- 基于样本特征的identity hash
- 增量更新机制

---

#### P3.2 元数据冲突解决

**问题描述**: 同一实体在不同数据库中的元数据不一致。

**示例**:
```
样本 SAMN12345678:
- NCBI: tissue = "brain", disease = "Alzheimer's disease"
- GEO: tissue = "cerebral cortex", disease = "AD"
- 融合后应该采用哪个值？
```

**策略维度**:
- 数据源权威性（CellXGene > GEO > NCBI?）
- 字段完整度
- 时效性
- 用户偏好

---

#### P3.3 融合结果的质量评估

**问题描述**: 如何量化融合结果的可信度。

**需要考虑的因子**:
- 原始数据源数量（多源验证 > 单源）
- 字段一致性（各源一致 > 存在冲突）
- 元数据完整度
- 用户反馈

---

### 4.5 P4: 系统架构问题

#### P4.1 渐进式查询策略

**问题描述**: 查询失败时如何优雅降级。

**当前策略**:
```
Level 1: Exact match (IN (...))
Level 2: LIKE pattern match
Level 3: 去除低优先级条件
Level 4: 全表扫描 + 提示
```

**待优化**:
- 基于用户反馈的动态策略调整
- 部分结果返回（"找到100条严格匹配，1000条近似匹配"）

---

#### P4.2 记忆与学习机制

**问题描述**: 从用户交互中学习，持续改进。

**三层记忆现状**:
- Working Memory: 会话级上下文 ✓
- Episodic Memory: 历史查询记录 ✓
- Semantic Memory: 成功查询模式 ✓

**待解决**:
- 从成功查询中提取可复用模式
- 从失败查询中识别知识缺口
- 个性化适配（用户A和用户B的"brain"可能指不同scope）

## 7. 与论文发表相关的关键创新点

### 7.1 差异化定位

相比于现有工作（SRAgent、CellAtria等），我们的核心创新在于：

1. **跨库统一查询**: 不仅发现数据，更提供统一的查询和融合视图
2. **本体感知语义扩展**: 不是简单的关键词匹配，而是基于生物医学本体的智能展开
3. **混合TAG范式**: Text2SQL + Ontology Expansion + 结果融合的三段式架构
4. **数据血缘透明化**: 完整的结果可追溯性和质量评估

### 7.2 需要重点突破的研究问题

- **RQ1**: 如何在保证查询精确性的前提下，有效处理umbrella term的语义扩展？
- **RQ2**: 如何在没有ground truth的情况下，评估跨库去重的准确性？
- **RQ3**: 如何设计渐进式查询策略，在LLM-free和LLM-augmented之间找到最佳平衡点？
