## 2. 核心优势

### 2.1 架构设计亮点

1. **Schema-Agnostic Tooling 原则**
   - 通过schema introspection动态适应，避免硬编码
   - 为未来数据库schema变化预留灵活性
   - 降低了维护成本

2. **规则优先+LLM兜底策略 (70/30)**
   - 成本意识强，减少不必要的LLM调用
   - 延迟敏感场景(如ID查询)可用规则快速响应
   - 符合生产系统对成本控制的要求

3. **三层记忆系统设计**
   - Working/Episodic/Semantic分层合理
   - 区分会话级、用户级、系统级数据生命周期
   - Schema知识库是V2的关键升级

### 2.2 技术选型合理性

| 组件 | 选型 | 优势 | 风险 |
|------|------|------|------|
| Agent框架 | 自研轻量框架 | 避免重型框架依赖，论文贡献清晰 | 需自行处理边界情况 |
| SQL生成 | 3候选+验证 | 借鉴CHASE-SQL最佳实践 | 增加单次查询延迟 |
| 本体集成 | 本地缓存 | 避免OLS API延迟和不稳定 | 需定期同步更新 |
| Web框架 | FastAPI | 原生async，自动OpenAPI文档 | Python生态依赖 |

## 3. 风险与问题

### 3.1 高风险项

**R1: LLM供应商锁定**
- **问题**: 深度依赖Claude的Tool Use功能
- **影响**: 切换LLM需重写部分交互逻辑
- **建议**: 抽象LLM接口层，支持多供应商无缝切换（非常重要）

**R2: 单Coordinator架构的扩展瓶颈**
- **问题**: 所有查询经过单一Coordinator
- **影响**: 高并发时可能成为瓶颈
- **建议**: 考虑Coordinator池化或横向扩展方案

### 3.2 中风险项

**M1: SQLite到PostgreSQL的迁移复杂度**
- 当前设计假设迁移只需改DAL，但实际情况可能更复杂
- 建议提前在开发环境搭建PG测试实例验证

**M2: 50K本体术语的查询性能**
- 本地缓存虽快，但复杂层级查询可能仍有性能压力
- 建议增加本体缓存的性能基准测试

### 4.1 架构层面

```
建议增加以下组件:
┌─────────────────────────────────────┐
│  Health Monitor (健康监控)          │

│  - 数据库连接池监控                  │
│  - 查询延迟告警                      │
├─────────────────────────────────────┤
│  Circuit Breaker (熔断器)           │
│  - LLM服务降级策略                   │
│  - 外部API失败保护                   │
├─────────────────────────────────────┤
│
└─────────────────────────────────────┘
```

### 4.2 技术选型优化

1. **LLM客户端**: 建议引入litellm等统一接口层
2. **缓存层**: 考虑增加Redis作为分布式缓存选项
3. **监控**: 增加OpenTelemetry tracing支持

### 4.3 实施路线调整建议

| 阶段 | 原计划 | 建议调整 |
|------|--------|---------|
| Phase 0 | 3-5天 | 保持，但增加技术选型验证 |
| Phase 1 | 7-10天 | 延长至10-12天，预留调优时间 |
| Phase 2 | 5-7天 | 保持 |
| Phase 3 | 7-10天 | 建议与Phase 2并行启动部分工作 |
| Phase 4 | 7-10天 | 保持，但用户研究需提前招募 |
